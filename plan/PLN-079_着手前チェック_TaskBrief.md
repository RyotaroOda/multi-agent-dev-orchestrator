# PLN-079 着手前チェック Task Brief

## Goal（目的）

- `PLN-079` を最優先タスクとして開始し、企画フェーズ制約の範囲で DEV-001 の実装着手前チェック（参照順・記録先・監視項目）を固定する。

## Non-goals（非目的）

- 実装コードの作成
- 依存追加
- ビルド/テスト実行

## Acceptance Criteria（受け入れ条件）

- `plan/バックログ.md` で `PLN-079` が `doing` に更新されている。
- `plan/実装進捗ダッシュボード.md` のスナップショット/クリティカルパス/Issue別進捗が `PLN-079=doing` と一致している。
- `plan/意思決定ログ.md` に着手判断（根拠/代替案/影響/見直し日）が記録されている。
- `R-033` の予防策に「着手前チェック開始」が反映されている。

## Constraints（制約・禁止事項）

- セキュリティ優先順位 `A:漏えい防止 > B:ホスト保護 > C:課金回避` を崩さない。
- 企画フェーズ中はドキュメント更新のみを行い、実装作業は開始しない。
- `run_id` / `app_slug` の監査追跡要件を維持する。

## Scope（変更してよい範囲）

- 対象ディレクトリ: `plan/`
- 対象ファイル:
  - `plan/PLN-079_着手前チェック_TaskBrief.md`
  - `plan/バックログ.md`
  - `plan/実装進捗ダッシュボード.md`
  - `plan/意思決定ログ.md`
  - `plan/リスク登録簿.md`

## How to Test（検証手順）

- 台帳整合確認:
  - `PLN-079` の状態が `バックログ` と `ダッシュボード` の双方で `doing` になっている。
  - ダッシュボードの件数内訳（`done/doing/ready/todo`）がバックログと一致する。
- 判断記録確認:
  - `意思決定ログ` の最新ADRに、企画フェーズ制約下での着手方針が明記されている。
- リスク確認:
  - `R-033` の予防策が今回の着手運用を含む内容へ更新されている。

## Risks（主要リスク）

- `PLN-079` を `doing` にした後、実装着手条件が曖昧なままだと状態だけ先行する。
- クリティカルパスの更新がダッシュボードとずれると、期限判断が再びセッション依存になる。

## Rollback（戻し方）

- `PLN-079` を `ready` に戻す場合は、同時に `ダッシュボード` と `意思決定ログ` の着手記録を巻き戻して整合を維持する。
